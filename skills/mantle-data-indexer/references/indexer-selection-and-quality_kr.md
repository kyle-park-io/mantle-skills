# 인덱서 선택 및 품질

이 가이드를 사용하여 인덱스 소스를 선택하고 신뢰도를 전달하세요.

## 도구 매핑 및 엔드포인트 가용성

- GraphQL 경로: `mantle-cli indexer subgraph --endpoint <url> --query <graphql> --json`을 사용하세요.
- SQL 경로: `mantle-cli indexer sql --endpoint <url> --query <sql> --json`을 사용하세요.
- 엔드포인트는 런타임 입력입니다; 없을 때 URL을 조작하지 마세요.
- E2E `endpoint-configured` 시나리오의 경우, `E2E_SUBGRAPH_ENDPOINT` / `E2E_SQL_ENDPOINT`가 설정되지 않았을 수 있으며 해당 시나리오는 건너뛰어야 합니다.

## 소스 선택

- 다음 경우 `GraphQL`을 선택하세요:
  - 유지 관리되는 서브그래프가 있는 경우.
  - 유연한 엔티티 필터링과 중첩 필드가 필요한 경우.
- 다음 경우 `SQL`을 선택하세요:
  - 대상 프로토콜에 대한 웨어하우스/인덱서 테이블이 있는 경우.
  - 무거운 집계 또는 윈도우 함수가 필요한 경우.

## 모든 답변에 필요한 메타데이터

- `source_type` (`graphql` 또는 `sql`)
- 엔드포인트 또는 데이터셋 이름
- 절대 UTC 타임스탬프의 시간 범위
- 쿼리 실행 시간 (`queried_at_utc`)
- 알려진 지연 상태 (있는 경우)

## 데이터 품질 검사

1. 체인 범위가 Mantle인지 확인합니다.
2. 타임스탬프 단위 (`seconds` vs `milliseconds`)를 확인합니다.
3. 소수점 정규화 가정을 확인합니다.
4. 페이지네이션이 데이터를 잘라내지 않았는지 확인합니다 (`hasNextPage=true` 경고는 후속 페이지 필요).
5. SQL 결과 잘라내기 (`truncated=true`)가 필요한 행을 숨기지 않았는지 확인합니다.
6. 정렬/순서 키가 결정론적인지 확인합니다.

## 지연 및 완전성 정책

- 지연이 알 수 없으면 `지연 상태 알 수 없음`이라고 명시하세요.
- 지연이 알려져 있고 무시할 수 없으면 영향 설명을 포함하세요.
- 쿼리가 중간에 실패하면 명확한 경계와 함께 부분 결과를 보고하세요.
- 도구 경고가 있으면 출력에 복사하고 영향을 설명하세요.

## 흔한 실패 모드

- 잘못된 네트워크 또는 잘못된 엔티티 ID로 쿼리가 0개 행을 반환하는 경우.
- 엔드포인트가 없을 때 에이전트가 URL을 추측하는 대신 요청합니다.
- 정규화되지 않은 토큰 소수점으로 인한 집계 불일치.
- 중첩된 페이지네이션 창으로 인한 이중 계산.
- 혼합된 시간 세분성으로 인한 오해를 유발하는 합계.
